# Паттерн: repository (порт в домене, адаптер в инфраструктуре)

Порт — трейт в `domain` (например `InviteRepository`), реализация — в
`infrastructure` на `sqlx` поверх `PgPool`. Композиция (какой адаптер под какой
порт) — только в `api`. Домен про sqlx не знает (ADR-0003).

## Как делаем

- Трейт-порт `#[async_trait] ...: Send + Sync`, методы возвращают `Result<_, RepositoryError>`.
- Адаптер хранит `PgPool` (дёшево клонируется через `Arc`), мапит строки в доменные типы.
- SQL явный, через runtime-API `sqlx` (`query`/`query_as`), без compile-time макросов —
  чтобы `cargo check`/CI были зелёными без БД. Проверка SQL — в интеграционных тестах против БД.
- Ошибки драйвера мапятся в `RepositoryError` (см. [[error-handling]]); детали sqlx наружу не текут.

## Реконституция агрегатов (gotcha)

У агрегатов приватные поля и конструкторы с инвариантами. Пока в домене нет
отдельных rehydrate-конструкторов, адаптер восстанавливает агрегат через
существующий API: `User::register(..) (+ verify())`, `Invite::issue(.., Unlimited ctx)`
(+ `accept()` для принятых). Это сознательный хак на время; правильные
reconstitute-конструкторы — кандидат в будущий рефактор домена (предложить ADR).

## Транзакции

Инвариант с конкуренцией (выдача инвайта) требует атомарности — порт единицы
работы + `SELECT ... FOR UPDATE` (ADR-0011). Решение принимает домен, транзакция — деталь адаптера.
