# ADR-0013: Аутентификация и сессии

## Status

Accepted — 2026-06-26

## Context

До этого решения у системы нет понятия «текущий пользователь»: HTTP-хендлеры
принимают `author`/`viewer` параметрами и слепо доверяют им. Это долг и дыра: писать
можно от чужого имени. Дальше по роадмапу — маркет с гейтом верификации (ADR-0010),
которому нужно достоверно знать, кто и верифицирован ли он. Значит, нужен слой
аутентификации: как юзер доказывает, что он — это он, и как это доказательство
переживает несколько запросов.

Рамки: сеть закрыта инвайт-гейтом (ADR-0005), почтовой инфраструктуры пока нет,
масштаб underground. Источник правды и чистота домена — ADR-0003.

## Options considered

- **Где живут учётные данные — расширить `identity::User`.** Нет: агрегат identity
  заморожен (ADR-0003, «domain — source of truth»), и «кто ты» (identity) — это не
  «как ты это доказываешь» (auth). Смешивать пароль/сессию с профилем-ролью —
  размывание границы.
- **Отдельный контекст `auth` (выбрано).** `Credential` (хэш пароля, 1:1 с юзером) и
  `Session` живут в новом контексте `auth`; identity не трогаем. Чистая граница: auth
  ссылается на `UserId`, не наоборот.
- **Сессии: JWT (stateless) vs серверные токены (выбрано серверные).** JWT нельзя
  отозвать без чёрного списка — logout становится фикцией. Серверная сессия (строка в
  БД, непрозрачный токен у клиента) тривиально гасится удалением строки, ревокация
  бесплатна, состояние мало. Для нашего масштаба stateless-выигрыша нет.
- **Вход: пароль vs magic-link/passwordless.** Passwordless требует e-mail/SMS —
  инфраструктуры нет. Пароль на старте; магик-линк/2FA — позже отдельным решением.
- **Хэш токена в БД: хранить сырой токен vs хэш токена.** На MVP — сырой токен под
  UNIQUE-индексом (проще). Захардкоженный риск: дамп БД = угон активных сессий.
  Будущее ужесточение (хранить только хэш токена) — кандидат в правку/новый ADR.

## Decision

**Контекст `auth` в `domain`.** Value objects `Password` (политика длины, Debug скрыт,
никогда не сохраняется), `PasswordHash` (непрозрачный результат хэшера), `SessionToken`
(непрозрачный высокоэнтропийный секрет, URL-safe, Debug скрыт). Агрегаты: `Credential`
(`UserId` + `PasswordHash`, один на юзера) и `Session` (`SessionId`, токен, юзер, выдан,
истекает; инвариант срока — `Session::is_active`). Порты `CredentialRepository` и
`SessionRepository`. Срок сессии по умолчанию — `SESSION_TTL` = 30 дней.

**Хэширование и генерация токена — на границе (ADR-0003).** Это тяжёлые
недетерминированные операции (случайная соль, CSPRNG), их место — порты `application`
(`PasswordHasher`, `SessionTokenFactory`), реализация — в `infrastructure` (argon2id для
паролей; CSPRNG + base64url для токенов). Домен получает готовый `PasswordHash`/
`SessionToken` и только хранит/валидирует — как уже сделано с кодом инвайта.

**Use-cases.** `EstablishCredential` (хэшировать пароль, сохранить креды),
`LogIn` (по handle+паролю: сверка → выпуск сессии, токен → клиенту),
`LogOut` (погасить сессию по токену, идемпотентно),
`Authenticate` (по токену → текущий юзер с `verified` для гейта ADR-0010). Логин не
различает «нет юзера»/«неверный пароль» — оба дают одну ошибку (анти-энумерация).

**Транспорт токена (HTTP-срез, шаг 2).** HttpOnly+Secure+SameSite кука для web и
`Authorization: Bearer` для api/mobile. Извлечение «текущего юзера» — экстрактор/слой
поверх `Authenticate`; параметры `author`/`viewer` из хендлеров убираются.

**Пароль при регистрации.** Регистрация по инвайту начинает требовать пароль; создание
кредов вшивается в транзакцию регистрации (атомарно с юзером и профилем) на HTTP-срезе.

## Consequences

Получаем: достоверный «текущий пользователь», ревокация через logout, основа для гейта
верификации (ADR-0010) и ленты закрытых групп; домен auth чист и тестируем без I/O.

Теряем / рискуем / откладываем:
- **Сырой токен в БД на MVP** — при дампе БД активные сессии угоняемы до истечения.
  Принято осознанно; хэширование токена при хранении — будущая правка.
- **Нет refresh/скользящего срока** — по истечении 30 дней нужен повторный логин.
- **Нет rate-limit на логин** (перебор пароля) — это забота шлюза/инфраструктуры;
  отдельное решение. Длина пароля сверху ограничена (защита от DoS на хэшировании).
- **Нет сброса/смены пароля и 2FA** — `EstablishCredential` переиспользуется под смену
  пароля позже; magic-link/2FA — отдельным ADR.
- Регистрация меняет форму (нужен пароль) и транзакцию — затрагивается на HTTP-срезе.
- **Bootstrap-админ `root`** (seed-миграция 0002) создаётся без кредов и потому не может
  залогиниться. argon2-хэш в SQL не посчитать. Решение отложено: одноразовая установка
  пароля root через env/CLI на старте — кандидат на отдельный шаг. Пока root-логин не
  нужен (выдача инвайта на срезе 2a ещё по параметру).
