# ADR-0011: Атомарность выдачи инвайта (транзакция + блокировка инвайтера)

## Status

Accepted — 2026-06-25

Принят при ревью Prompt 2. Закрывает TOCTOU, помеченный в коде `IssueInvite::execute`.

## Context

Доменный инвариант (≤2 активных, кулдаун ~12ч — ADR-0005) корректен в `Invite::issue`,
но прикладной use-case читает счётчик активных и время последней выдачи отдельными
запросами, затем сохраняет — неатомарно. Два конкурентных запроса для одного инвайтера
оба прочитают `active = 1`, оба пройдут проверку и сохранят → 3 активных. Инвариант
силён ровно настолько, насколько консистентно чтение.

## Options considered

- **Ничего (best-effort).** Просто, но инвариант обходится под гонкой. Цена — лишние инвайты, тихая эрозия гейта. Отклонено.
- **Декларативный constraint «≤2 активных».** В Postgres «не больше N строк на ключ» не выражается обычным `UNIQUE`; нужен триггер или `EXCLUDE` с хаком. Неявно и хрупко. Отклонено.
- **Advisory lock без общей транзакции.** Лок и запись на разных соединениях — между ними та же гонка. Отклонено.
- **Транзакция + `SELECT ... FOR UPDATE` на строке инвайтера (выбрано).** Блокировка строки инвайтера сериализует его выдачи; чтение счётчика, доменное решение `Invite::issue` и вставка идут в одной транзакции.

## Decision

`IssueInvite` выполняется в одной транзакции; строка инвайтера блокируется
`SELECT ... FOR UPDATE`. В `application` вводится порт единицы работы (транзакции),
дающий tx-привязанные чтения/запись; решение по-прежнему принимает домен
(`Invite::issue`), атомарность — деталь `infrastructure`. Композиция — в `api`.

## Consequences

Получаем: инвариант ADR-0005 держится под конкуренцией, гарантия не слабее доменного
правила. Теряем/усложняем: выдачи одного инвайтера сериализуются (не проблема — они
редки, кулдаун 12ч); в `application` появляется транзакционный порт, в `infrastructure`
— его реализация; use-case `IssueInvite` из Prompt 1 переписывается под порт, его юнит-тесты
обновляются. Приём инвайта (`AcceptInvite`) тоже оборачивается в транзакцию для целостности.
