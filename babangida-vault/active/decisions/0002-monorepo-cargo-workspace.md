# ADR-0002: Монорепо на cargo workspace

## Status

Accepted — 2026-06-25

## Context

Стек — несколько крейтов на Rust (`shared`, `domain`, `application`, `infrastructure`, `api`, `uikit`, `web`, `mobile-ios`, `mobile-android`) с жёстким направлением зависимостей внутрь. Команда соло/маленькая, релизятся слои синхронно. Рядом с кодом живёт vault со знаниями и решениями (`babangida-vault/`).

## Options considered

- **Polyrepo (крейт = репо).** Плюс: формальная изоляция, отдельные права доступа. Минус: изменение, проходящее через слои (`domain` -> `application` -> `api`), разваливается на N синхронизированных PR в N репозиториях, версионирование через git-теги и `[patch]`. Для одного человека это чистый оверхед без выгоды.
- **Один большой крейт с модулями.** Плюс: максимально просто, одна цель сборки. Минус: границы слоёв держатся только на честном слове — `domain` ничто не мешает потянуть `sqlx` или `axum`. Ровно то, что мы хотим запретить на уровне компилятора.
- **Монорепо с cargo workspace.** Один репо, один `Cargo.toml`-workspace, крейты — прямые дети корня. Границы зависимостей становятся фактом графа крейтов, а не договорённостью. Атомарный коммит через все слои, один lock-файл, один CI.

Отдельно — где живёт vault:

- **Отдельный репо для vault.** Плюс: чистое разделение «код» / «знания». Минус: решения и код расходятся во времени, ADR описывает не тот коммит, две истории на одно изменение.
- **Vault в том же репо (`babangida-vault/`).** ADR и код, который он описывает, едут в одном коммите. Решение и его реализация всегда консистентны.

## Decision

Монорепо на cargo workspace, крейты — прямые дети корня репо. Vault лежит рядом, в `babangida-vault/`. Границы между слоями обеспечивает граф зависимостей workspace, а атомарные изменения через слои и единый CI снимают главную боль соло-разработки.

## Consequences

Получаем: атомарные изменения через слои в одном PR, один lock-файл и единый CI, границы крейтов, проверяемые компилятором, консистентность ADR с кодом.

Теряем и усложняем: время сборки растёт вместе с workspace — спасают инкрементальная компиляция и `cargo build -p`, но «холодный» CI будет ощутимо дольше. Дисциплина границ перекладывается на нас: cargo не запретит лишнюю зависимость в `Cargo.toml`, направление внутрь придётся защищать ревью или линтом. Отдельные права доступа на крейт невозможны — весь репо или ничего.
