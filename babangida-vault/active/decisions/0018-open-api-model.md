# ADR-0018: Модель открытого API (персональные ключи)

## Status

Accepted — 2026-06-27

## Context

ADR-0010 ввёл верификацию как гейт привилегий; открытое API — третья и последняя
такая зона (после маркета и музыки). Соц-ядро, верификация и музыка готовы; пора дать
программный доступ к сети для интеграций и power-users. Нужна модель: как
аутентифицировать вызовы `/api/v1`, кто может получить доступ, что API отдаёт и как это
согласуется с анти-ВК (не отдельный продукт, а программный доступ к той же сети) и с
источником правды в домене (ADR-0003).

## Options considered

- **Аутентификация: персональные ключи vs OAuth2-приложения vs переиспользование сессии
  (выбрано персональные ключи).** OAuth2 (регистрация приложений, authorization-code,
  scoped/refresh-токены, экран согласия) — большой отдельный проект, избыточный для MVP.
  Сессионный токен — короткоживущий и login-scoped, не для программного/стороннего
  доступа, без отзыва и меток. Персональные ключи (как personal access token у GitHub) —
  простой стандартный путь: владелец выпускает именованный отзываемый ключ, интегратор
  шлёт его как Bearer.
- **Гейт верификации в домене (по образцу маркета/музыки).** Право выпускать ключи —
  инвариант контекста openapi; держим в домене ([`ApiKey::issue`] требует
  `VerifiedStatus`), `application` читает статус.
- **Хранение секрета: сырой vs хэш (выбрано хэш SHA-256).** Сессии на MVP хранили токен
  сырым (ADR-0013, хэш — будущее). Но API-ключи долгоживущие и сторонние — храним хэш.
  SHA-256 (не argon2): ключ высокоэнтропийный, медленный хэш не нужен, а поиск по хэшу —
  на каждом запросе `/api/v1` (нужен O(1)). Сырой секрет показывается владельцу один раз.
- **Поверхность API: только чтение vs чтение + запись (выбрано чтение + запись).**
  Чтение — лента/профиль/музыка/маркет. Запись — пост и релиз трека от имени владельца
  (релиз — за тем же гейтом верификации). Гранулярные scope'ы (read-only ключи) —
  будущее.
- **Версионирование.** Префикс `/api/v1` с самого начала — чтобы менять контракт, не
  ломая интеграции.

## Decision

**Контекст `openapi` в `domain`.** Агрегат `ApiKey`: `owner`, `label` (`ApiKeyLabel`,
1..=80), `hash` (`ApiKeyHash`, непрозрачен), `status` (`active | revoked`). Секрет —
`ApiKeyToken` (Debug скрыт, генерится и хэшируется на границе). Выпуск —
`ApiKey::issue(..., verified, ...)`: при `casual` → `OpenApiError::NotVerified`. Отзыв
(`revoke`) — только владелец, только из действующего (`NotOwner`/`AlreadyRevoked`).
Порт `ApiKeyRepository` (`find_by_id`/`find_by_hash`/`save`; перечисление — read-модель).

**application.** Порты `ApiKeyFactory` (CSPRNG) + `ApiKeyHasher` (SHA-256). Use-cases
`IssueApiKey` (читает статус, домен решает; возвращает сырой токен один раз),
`RevokeApiKey`, `AuthenticateApiKey` (хэш → поиск → активен → владелец; иначе
`Unauthenticated`). Read-модель `ApiKeysOf`. `ApplicationError` += `OpenApi`.

**infrastructure/api.** Миграция `0009` (`api_keys`, `key_hash UNIQUE`).
`RandomApiKeyFactory` (`bbg_` + 40 алфанум), `Sha256ApiKeyHasher`. Управление под
сессией: `POST /api-keys`, `GET /api-keys`, `POST /api-keys/{id}/revoke`. Экстрактор
`ApiCaller` (Bearer-ключ → владелец). `/api/v1`: чтение (`me`/`feed`/`profiles/{handle}`/
`music`/`market`) и запись (`posts`/`tracks`) от имени владельца. Маппинг:
`NotVerified`/`NotOwner`→403, `AlreadyRevoked`→409, неизвестный/отозванный ключ→401.

**Анти-ВК.** Открытое API — программный доступ к той же сети (те же read-модели и
use-cases), не отдельный продукт; управление ключами — экран в той же сети.

## Consequences

Получаем: третью привилегированную зону за гейтом; программный доступ (чтение+запись)
к сети; ключи хэшируются (надёжнее сырых сессий); версионированный контракт `/api/v1`.

Теряем / откладываем:
- **Нет гранулярных scope'ов** — ключ действует от имени владельца целиком (чтение и
  запись). Read-only/ограниченные scope'ы — будущая работа.
- **Нет rate-limiting** — ключ не ограничивает частоту; защита от абьюза только гейтом
  верификации. Лимиты/квоты — отдельное решение.
- **Нет OAuth/сторонних приложений** — только персональные ключи; делегирование доступа
  приложениям без передачи ключа — будущее.
- **Нет `last_used`/аудита** — не показываем, когда ключ использовался последний раз.
- **Слабая инвалидация при де-верификации** — модель не снимает верификацию, так что
  ключи верифицированного остаются валидны; при появлении де-верификации это надо будет
  увязать.
