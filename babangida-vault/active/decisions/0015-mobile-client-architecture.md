# ADR-0015: Архитектура mobile-клиента

## Status

Accepted — 2026-06-27

## Context

PoC mobile (Prompt 4, ADR-0007) доказал шеринг `domain`/`shared`/`uikit`: один
экран-лента из доменных значений, статичные данные, без сети. Шелл `App` дублируется
в `mobile-ios` и `mobile-android`. Дальше — основная разработка: реальные экраны с
данными из HTTP-API и навигацией. Web берёт данные через Leptos server-функции (SSR),
у mobile SSR нет — это клиент, который ходит в API напрямую. Нужно зафиксировать, как
mobile получает данные, хранит сессию и навигирует, и где живёт общий UI-код.

## Decision

1. **Общий крейт `mobile-ui`.** Dioxus-экраны, навигация и data-client — в одном
   крейте; `mobile-ios`/`mobile-android` становятся тонкими шеллами (только нативный
   `launch` под фичей `shell`, делегируют в `babangida_mobile_ui::App`). Снимает
   дублирование шелла, отмеченное в PoC. Домен/шелл-граница из ADR-0007 сохраняется:
   `mobile-ui` зависит от `uikit`/`domain`/`shared`, рендерер включают только шеллы.
2. **Данные — прямой HTTP через `reqwest`.** Mobile зовёт тот же API, что и web, но
   напрямую (без SSR/server-функций — те только Leptos). Async-запросы — в
   `use_resource`/async-обработчиках Dioxus; рантайм даёт нативный шелл (`dioxus/mobile`
   на tokio). DTO дублируются по-минимуму на стороне клиента (у web свои, у mobile свои)
   — общий DTO-крейт пока не вводим (преждевременная абстракция; кандидат, если
   дублирование начнёт болеть).
3. **Сессия — в памяти (MVP).** Токен сессии (ADR-0013) держим в сигнале и шлём как
   `Authorization: Bearer`. Безопасное хранилище (iOS Keychain / Android Keystore) и
   персист между запусками — отложено. Домен/шелл-граница чистая.
4. **Навигация — сигнальная (MVP).** Текущий экран — сигнал-enum в контексте, без
   `dioxus-router`. Роутер добавим, когда появятся deep links / реальная история; пока
   набор экранов мал и плоский, лишняя зависимость не нужна.
5. **Только тёмная тема на mobile (пока).** Как и в PoC: токены тёмной темы инлайном;
   светлая/переключатель — после стабилизации (на web уже есть, ADR-0008).

## Consequences

- Получаем: основной mobile-UI и data-client в одном месте, шеллы тонкие, дублирования
  шелла больше нет; mobile ходит в тот же API, что web — единый контракт.
- Теряем/рискуем: дублирование DTO web↔mobile; токен в памяти (после рестарта нужен
  повторный вход); `reqwest`+рантайм на незрелом mobile-тулчейне — часть риска ADR-0007
  (на устройстве не проверено, только хостовая компиляция UI-ядра и фичи `shell`).
- Становится сложнее: при росте экранов сигнальная навигация упрётся в потолок —
  тогда миграция на `dioxus-router`; общий DTO-крейт — при росте дублирования.
- Митигация: граница домен/шелл держится чистой (план Б ADR-0007 стоит только UI);
  data-client изолирован в модуле, переезд на роутер/секьюр-стор локален.
