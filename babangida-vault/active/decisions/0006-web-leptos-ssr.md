# ADR-0006: Web на Leptos в режиме SSR

## Status

Accepted — 2026-06-25

## Context

Нужен web-клиент для закрытого комьюнити: ленты, профили, маркетплейс. Команда соло/маленькая, бэкенд на Rust. Хочется не плодить второй язык и второй набор инструментов на фронте без серьёзной причины. Доступ по инвайтам, но публичные страницы (лендинг, профили) должны нормально индексироваться и быстро отдавать first paint.

## Options considered

- **Leptos CSR/SPA.** Чистый Rust, тот же стек. Но первый рендер — пустой `<div>` + загрузка wasm, плохой first paint и нулевой SEO без отдельного пререндера.
- **Leptos SSR (SSR + гидратация).** Тот же Rust, сервер отдаёт готовый HTML, потом гидратация. Хороший first paint и SEO. Минус — гидратация добавляет сложность (двойной рендер, расхождения server/client), Leptos молод.
- **Yew.** Зрелее по возрасту, но в основном CSR-история, та же проблема с first paint, и реактивность менее удобная, чем сигналы Leptos.
- **JS-фреймворк (Next.js/SvelteKit).** Огромная экосистема, готовые компоненты. Но это второй язык, второй тулчейн, и UI больше нельзя шарить с mobile-крейтами на Rust.
- **Серверный HTML (Axum + Askama + htmx).** Просто и быстро для контентных страниц, минимум клиентского кода. Но для интерактивного приложения (ленты, плеер, реалтайм) упрёмся в потолок htmx.
- **Phoenix LiveView.** Сильная модель для реалтайма, но тянет Elixir/BEAM — третий рантайм рядом с Rust-бэком, чего соло-команда не потянет.

SSR, а не SPA — ради first paint и SEO публичных страниц; пустой экран до загрузки wasm для соцсети неприемлем. Leptos, а не JS — чтобы фронт и бэк жили на одном Rust, делили `shared` и `uikit`, и не дробить и без того маленькую команду на два языка.

## Decision

Web-клиент — **Leptos 0.7 в режиме SSR** (server-side rendering + гидратация). Один язык на всём стеке, общий код и `uikit` с mobile, и при этом приличный first paint и индексация.

## Consequences

Получаем единый Rust-стек, переиспользование `shared`/`uikit`, быстрый первый рендер и работающий SEO.

Теряем и усложняем:

- Экосистема Leptos меньше React/Svelte — готовых компонентов и решений придётся писать самим.
- Гидратация — источник класса багов: server и client должны рендерить одинаково, иначе расхождения.
- Leptos молод, API ещё двигается; обновления минорных версий могут ломать.
- SSR-сервер — ещё один рантайм-компонент, который надо держать живым, а не просто отдавать статику с CDN.
