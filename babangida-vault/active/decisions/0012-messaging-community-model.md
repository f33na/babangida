# ADR-0012: Модель messaging и сообществ (DM, группы/паблики)

## Status

Accepted — 2026-06-26

## Context

Prompt 5 вводит личную переписку и сообщества. Нужны явные границы: что такое личный диалог, что такое группа и паблик, кто может писать, читать и вступать. Два требования формируют рамку. Анти-ВК: фичи живут в общей ленте/профиле/нави, а не отдельными приложениями. ADR-0010: переписка и базовые соцфункции доступны без верификации (верификация гейтит только маркет/музыку/API).

## Options considered

- **Messaging — DM только между взаимными контактами.** Точнее по приватности, но графа дружбы в продукте пока нет; вводить его только ради DM — преждевременно. Отклонено на старте.
- **Messaging — любой участник пишет любому (выбрано).** Сеть и так закрыта инвайт-гейтом; этого барьера достаточно для старта. Граф контактов и настройки приватности DM — позже, если появится спам/абьюз.
- **Групповые чаты внутри messaging.** Нет: у группового общения другие инварианты (членство, роли, модерация) — это контекст community, не messaging.
- **Сообщества — один тип «группа».** Мало для underground: нужны и приватные тусовки, и вещание на аудиторию. Отклонено.
- **Сообщества — closed + public (выбрано).** `closed` — читают/пишут только участники, вступление по приглашению модератора (ощущение приватного чата); `public` — читают все, публикуют только модераторы, вступление свободное (ощущение вещания). Две потребности — одной моделью.
- **Отдельный контент-силос для сообществ.** Нарушает анти-ВК. Контент сообществ должен течь в общую ленту (контекст `content`), а не жить отдельно.

## Decision

**messaging.** DM — это `Conversation` двух участников; пара канонизируется (меньший UUID первым), поэтому `(a, b)` и `(b, a)` — один диалог. Единственный инвариант записи: автор сообщения обязан быть участником диалога (`Conversation::send`). Любой участник может открыть диалог с любым. Без верификации (ADR-0010). Уникальность пары и атомарность создания диалога — забота `infrastructure` (индекс), не домена.

**community.** Сообщество = `closed | public`, роли `owner | moderator | member`. Инварианты — в агрегате `Group`: основатель становится владельцем; всегда есть хотя бы один владелец (нельзя снять/вывести последнего); самостоятельно вступить можно только в `public`, в `closed` участника добавляет модератор; владельца не убирают и не назначают рядовыми действиями. Право публикации: в `closed` — любой участник, в `public` — только модераторы. Чтение: `public` — все, `closed` — только участники.

**Анти-ВК.** Контент сообществ идёт в общую ленту (контекст `content`), не отдельным приложением. Связь поста с сообществом и гейт публикации через `Group::can_post` реализуются на HTTP-срезе (следующий промт) без правки замороженного агрегата `content::Post`.

## Consequences

Получаем: ясные границы DM и сообществ; единая модель для приватных тусовок и пабликов; инварианты владельца и прав сосредоточены в домене (`Group`), хендлеры их не дублируют.

Теряем / рискуем / откладываем:
- Нет графа контактов и настроек приватности DM — любой пишет любому. Если появится абьюз, добавим граф/блокировки отдельным решением.
- Членство держится внутри агрегата `Group` (просто и целостно для небольших сообществ). Для очень крупных пабликов это пересматривается на отдельный membership-агрегат/проекцию — кандидат в будущий ADR.
- Атомарность вступления/смены ролей под конкуренцией (гонка на инварианте последнего владельца) решается на `infrastructure` блокировкой строки группы — по образцу выдачи инвайта ([[decisions/0011-invite-issuance-atomicity]]); делается на HTTP-срезе.
- Связь пост↔сообщество добавляется в `content` на следующем промте без изменения доменного агрегата `content::Post` (реконституция/ссылка на границе).
