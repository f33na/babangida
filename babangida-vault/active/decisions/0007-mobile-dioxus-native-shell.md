# ADR-0007: Mobile на Dioxus с нативным шеллом

## Status

Accepted — 2026-06-25

## Context

Нужны мобильные клиенты под iOS и Android для соцсети по инвайтам. Команда соло/маленькая, переписывать домен и UI трижды (web, iOS, Android) нет ресурсов. Бэкенд и web уже на Rust, есть `domain`, `shared` и `uikit`, которые хочется переиспользовать. Это самая рискованная ставка проекта.

## Options considered

- **Dioxus mobile (`mobile-ios`, `mobile-android`).** Rust везде, шеринг `domain`, `shared` и `uikit` с web почти как есть. Один язык, одна модель крейтов, нет дублирования бизнес-логики. Минус — незрелость: Dioxus mobile сырой, тулчейн сборки под iOS/Android нестабилен, рендеринг и нативные интеграции (пуши, камера, прохождение стор-ревью) могут просто не взлететь.
- **React Native.** Зрелый, огромная экосистема, понятный путь в сторы. Минус — JS/TS-стек, домен и UI пришлось бы дублировать или прокидывать Rust через FFI/uniffi, что съедает весь выигрыш от шеринга.
- **Flutter.** Зрелый рендер, хорошая производительность, единый кодбейс под обе платформы. Минус — Dart, ещё один чужой стек, Rust только через FFI; снова дублирование логики и второй язык в проекте.
- **Нативка (Swift + Kotlin).** Максимум контроля, лучший UX, без вопросов к сторам. Минус — две отдельные кодовые базы, два языка, для соло-команды это утроение работы относительно Rust-пути.
- **Capacitor/PWA.** Дёшево и быстро завернуть web. Минус — слабые нативные возможности, плохой доступ к камере/пушам, риски при ревью в App Store, не тот UX для медиа-приложения.
- **Tauri mobile.** Тоже Rust, но UI на webview, а не нативный рендер. Минус — mobile-направление у Tauri тоже раннее, а webview-подход даёт те же ограничения, что и PWA.

## Decision

Берём Dioxus 0.6 для `mobile-ios` и `mobile-android`: нативный шелл вокруг Rust-ядра, общий с web `uikit` и `domain`/`shared`. Причина — один язык на весь проект и реальный шеринг кода критичны для маленькой команды; ради этого сознательно идём на самый высокий риск в проекте.

План Б зафиксирован: если Dioxus mobile не вытянет (рендер, тулчейн сборки, пуши/камера или прохождение стор-ревью), переходим на нативку (Swift + Kotlin) либо React Native, оставляя `domain`/`shared` под капотом через FFI/uniffi. UI-слой (`uikit`) при этом теряется, домен — нет.

## Consequences

- Получаем: один язык на всю кодовую базу, переиспользование `domain`, `shared` и `uikit`, отсутствие дублирования бизнес-логики между web и mobile.
- Теряем/рискуем: завязка на незрелый Dioxus mobile и сырой тулчейн iOS/Android; рендер и нативные интеграции (пуши, камера, ревью в сторах) могут не заработать и потребовать смены курса.
- Становится сложнее: отладка проблем сборки и нативных мостов, оценка сроков (неизвестно, где упрёмся), возможная вынужденная миграция UI на нативку/React Native с потерей `uikit`.
- Митигация: ранний прототип под обе платформы до основной разработки, чтобы проверить рендер и стор-ревью; держим `domain`/`shared` чистыми от Dioxus, чтобы план Б стоил только переписывания UI-слоя.
