# ADR-0004: CQRS: разделение команд и запросов

## Status

Accepted — 2026-06-25

## Context

`domain` хранит инварианты модели: лимиты инвайтов, кулдаун, права верифицированных юзеров. Если те же доменные типы тащить в чтение, каждый экран начинает грузить агрегаты целиком и реконструировать инварианты ради того, чтобы показать ленту. Запись и чтение имеют разную форму нагрузки и разные требования к данным.

## Options considered

- **Единая CRUD-модель.** Одни типы на чтение и запись. Просто и привычно, меньше кода. Минус: read-модели тянут доменные инварианты в запросы, экраны вынуждены работать через агрегаты, появляются лишние загрузки и неуместная логика в путях чтения.
- **Лёгкий CQRS (разные модели, одна БД).** Commands проходят через `domain` и его инварианты; queries бьют в read-модели, заточенные под конкретные экраны. Та же PostgreSQL. Плюс: чтение не зависит от формы агрегатов, запись остаётся под защитой домена. Минус: две модели вместо одной, дублирование части полей.
- **Полный CQRS + event sourcing + отдельное read-хранилище.** Максимальная гибкость и аудит. Минус: для соло/маленькой команды на старте это переусложнение — проекции, ресинк, eventual consistency, отдельная инфраструктура чтения ради нагрузки, которой пока нет.

## Decision

Берём лёгкий CQRS: разные модели для команд и запросов, одна PostgreSQL. Запись идёт через `domain`-инварианты, чтение — через read-модели под экраны. Это осознанно урезанный CQRS: без event sourcing и без отдельной БД для чтения, потому что domain-first защита нужна сейчас, а распределённое чтение — нет.

## Consequences

Получаем: чистый путь записи под защитой домена и быстрое чтение без реконструкции агрегатов. Теряем: единственный источник типов — теперь команда и запрос описываются раздельно, часть полей дублируется. Сложнее: при изменении схемы надо синхронно править и команды, и read-модели; легко забыть одну из сторон. Граница не event-sourced, поэтому read-модели читают те же таблицы — отдельной согласованности проекций нет, и это сознательно.
