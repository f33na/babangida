# Shubin Framework для babangida — гайд для агента

Этот файл учит тебя (Claude Code / ultracode и любого AI-агента) фреймворку,
по которому устроен проект babangida. Прочитай его один раз в начале работы над
репозиторием — дальше держи в голове.

Иерархия контекста, чтобы не путаться:

- **этот файл (`docs/shubinframework.md`)** — _понять_ фреймворк и его адаптацию под babangida. Меняется редко.
- **`babangida-vault/COMMON.md`** — короткий операционный чек-лист на каждую сессию. Читаешь перед кодом.
- **per-crate `AGENTS.md`** — указывает на `COMMON.md` и добавляет правила конкретного крейта.

Если этот файл и `COMMON.md` расходятся — прав `COMMON.md` (он ближе к коду). Заметил расхождение — скажи, не угадывай.

---

## 1. Зачем это вообще

Маленькая команда с AI-агентами упирается в две боли, и это одна и та же боль:

- **Амнезия агента.** Каждая сессия начинается с нуля. То, что решили вчера, завтра надо объяснять заново.
- **Потеря контекста человеком.** Через три месяца никто не помнит, почему выбран именно этот подход.

Обе — потому что контекст живёт в головах, а не в читаемой форме.

Решение: весь важный контекст переезжает в **единый markdown-репозиторий (vault)**, который одинаково читают и люди, и агенты. Что не записано — для нового агента и нового человека не существует.

Markdown-в-git выигрывает у Notion/Confluence по простой причине: агенты читают и пишут его нативно (без API и токенов), `grep` работает мгновенно, версионирование из коробки, нет vendor lock-in.

## 2. Два принципа (всё остальное — следствие)

**Принцип 1: append-only для истории, mutable для состояния.**
Лог решений, фидбэк, инциденты — только дописываются. Roadmap, страницы фич, статусы ADR — меняются. Это даёт честную историю без хаоса.

**Принцип 2: один source of truth на факт.**
У каждого решения, фичи, компонента ровно один канонический дом. Дубликаты расходятся: если одно и то же описано в трёх местах — через месяц получишь три разных описания.

Других принципов нет.

---

## 3. Source of truth в babangida (читай внимательно)

Это главная адаптация фреймворка под этот проект. Принцип 2 здесь делит ответственность жёстко:

- **Типы, модели, инварианты → крейт `domain` (код).** Это source of truth. Vault их НЕ дублирует. Хочешь узнать структуру `Invite` или `Profile` — смотри `crates/domain`, а не глоссарий.
- **Решения и контекст → vault.** Почему 2 инвайта и кулдаун 12ч, почему Leptos, почему CQRS — это ADR, glossary и patterns.

Глоссарий описывает _термины_ (ubiquitous language: invite, verified, public, casual, skin), а не _типы_. Типы — в коде.

Направление зависимостей (clean architecture) тоже неизменно:

```
domain ⟵ application ⟵ infrastructure / api / web / mobile
```

`domain` не зависит ни от чего внешнего — ни от axum, ни от sqlx, ни от leptos. Зависимости направлены только внутрь. Нарушить это направление — серьёзная ошибка, даже если «так быстрее».

## 4. Структура vault

Полная форма (babangida многокомпонентный, поэтому не lite):

```
babangida-vault/
├── stable/      редко меняется, source of truth для «почему»
├── active/      основная работа: features/, decisions/, patterns/
├── log/         append-only: feedback/, incidents/, retrospectives/
├── COMMON.md    операционная схема для агентов
└── index.md
```

Что живёт в каждой зоне:

- **`stable/`** — `stack.md`, `glossary.md`, `architecture.md`, `design-system.md`, `api-style.md`.
- **`active/features/`** — по папке на bounded context, внутри `index.md` (см. §5).
- **`active/decisions/`** — ADR, сквозная нумерация (см. §5).
- **`active/patterns/`** — rust-паттерны и `gotchas.md` (см. §7).
- **`log/`** — датированные файлы фидбэка, постмортемы, ретро. **Не редактируется** (append-only).

Папки внутри этих трёх зон добавляются, когда реально накопилось 2–3 файла под них. Не раньше. Пустые папки «на будущее» не создаём.

## 5. Единица работы: Feature + ADR

### Фича как каноническая страница

Каждая фича — `active/features/<slug>/index.md`. Это единица работы, не тикет. Тикет (если появится) ссылается сюда, не наоборот.

```md
---
status: in-progress       # idea | validated | in-progress | shipped | killed
owner: @fen
priority: P1              # P0 | P1 | P2 | P3
created: 2026-06-25
target: 2026-07-10
---

# <Название фичи>

## Why

Одно предложение: зачем и кому.

## Scope

Что строим. Границы: что НЕ делаем в этой итерации.

## Success criteria

Измеримая проверка готовности.

## Technical

Ссылки на [[decisions/000N-...]].

## Implementation

PR: #...

## Log

- 2026-06-25: создано
```

### ADR (Architectural Decision Record)

ADR пишется для решения, которое: трудно откатить, затрагивает несколько частей системы, придётся объяснять через три месяца.

Эвристика границы: **откатывается за день работы одного человека — ADR не нужен**. Откат — неделя и больше — нужен. Норма: 5–15 ADR за первые полгода, не 40. ADR на выбор HTTP-клиента — это bikeshedding, не ADR.

Формат (строго):

```md
# ADR-000N: <короткий заголовок решения>

## Status

Accepted — YYYY-MM-DD

# либо: Superseded by ADR-000M

## Context

Ситуация и ограничения.

## Options considered

Варианты с плюсами/минусами. Отклонённые тоже — через полгода вспомнишь, почему не пошёл туда.

## Decision

Выбранный вариант + предложение-два почему.

## Consequences

Что теряем, что получаем, что становится сложнее.

## Related

Ссылки на другие ADR, на фичу, на Issue (если есть).
```

ADR **иммутабелен после принятия**. Чтобы изменить — пишешь новый ADR со статусом `Supersedes ADR-000N`, а старому ставишь `Superseded by ADR-000M`. Не переписываешь старый.

### ADR-first — идеал, к которому стремимся

В идеале значимое решение начинается с PR, меняющего только ADR: обсуждение → принятие → код. На практике в соло часто решение принимается в голове, а ADR пишется после кода. Это нормально. Важно, что ADR в итоге существует.

## 6. Цикл фичи

Полный путь от идеи до прода, каждый шаг даёт артефакт в конкретном месте:

```
idea → spec (Why+Scope) → ADR (если нужен) → design → impl → preview/review → merge/deploy → close → feedback
```

- **idea/spec** живут в `index.md` фичи, статус двигается `idea → validated`.
- **ADR** появляется на шаге 3, только если есть нетривиальное решение на >1 части системы.
- **impl** — код в репозитории, ссылка на PR в секции Implementation.
- **close** — статус `shipped`, запись в Log.
- **feedback** — через время файл в `log/feedback/` с цифрами и выводами; если нужны изменения — новая фича.

Когда реальность расходится с планом (на impl понял, что подход из ADR не работает): не меняй код молча. Обнови старый ADR (`Superseded by`), создай новый с новым решением и причиной отказа от старого, поправь `index.md` если поменялся scope. Это 15 минут — и через полгода никто не чешет голову, почему код говорит одно, а ADR другое.

## 7. Как ты работаешь

### Перед тем как писать код

1. Найди фичу в `active/features/<slug>/index.md`.
2. Проверь `active/decisions/` на релевантные ADR.
3. Проверь `active/patterns/` на похожие паттерны и `gotchas.md`.

### Паттерны

`active/patterns/` накапливает решения типовых задач babangida: `error-handling.md`, `repository.md`, `specification.md`, `value-objects.md`, `cqrs.md`, `axum-handlers.md`, `leptos-components.md`, `dioxus-components.md`, `gotchas.md`.

Паттерн рождается **из реального решения**, а не из общих best practices. Первый раз — просто код. Второй — «возможно, паттерн». Третий — точно паттерн, записывай. `gotchas.md` пополняешь сразу после того, как разобрался со странным багом, пока свежо.

### Что ты делаешь

Генеришь код из спецификации, читаешь vault и отвечаешь в контексте, пишешь черновики ADR по запросу, делаешь рутинные изменения.

### Чего ты НЕ делаешь

Не принимаешь продуктовые решения. Не действуешь в проде без человека. Не пишешь ADR без ревью человеком (иначе получаются галлюцинированные trade-offs). Не коммитишь в main без PR и ревью.

### На значимое решение или gotcha

Предложи ADR (формат §5) или строку в `patterns`. **Файлы молча не создавай** — сначала скажи, что нашёл, и спроси.

## 8. Анти-паттерны (как ломают этот фреймворк)

- **Vault как write-only.** Самый частый провал: доки пишутся, но их никто не читает. Через 3 месяца — кладбище. Противоядие: ссылки на vault в каждом значимом PR, «а что vault говорит про это?» в обсуждениях.
- **AI генерит, никто не ревьюит.** ADR по запросу без ревью → через полгода половина ADR с выдуманными обоснованиями. ADR проходит ревью как любой PR.
- **Инфляция ADR.** 40 ADR за квартал, половина про мелочи. Держись эвристики «откатывается за день — не ADR».
- **Идеальный vault в первый месяц.** Две недели настройки структуры до первой строки кода. Начинай с трёх зон, расширяй по требованию.
- **Дорогой старт.** Kubernetes/микросервисы/Kafka в команде из одного. Бюджет сложности исчерпан до первой бизнес-логики. Начинай с модульного монолита на managed-сервисах.

## 9. Стиль документов

- Проза вместо булит-листов, где это разумно. Булиты — для настоящих перечислений, не для разбивки предложения по строкам.
- Кратко. Две строки вместо абзаца. Цель — чтобы ты и человек быстро ориентировались, а не «красиво задокументировано».
- Бэктики для имён файлов, путей и идентификаторов.
- Без эмодзи в канонических документах.
- Голос прямой и честный про trade-offs. Без корпоративного хеджирования и AI-воды («более того, важно отметить, что...»).
- Язык vault — выбран русский (как `COMMON.md`). Не смешивай: либо весь vault русский, либо весь английский. Один подход.

## 10. Что НЕ делаем сейчас

babangida — соло/маленькая команда на старте. Фреймворк прямо советует усложнять по реальной боли, а не по плану:

- **Без ритуалов.** Daily-standup с самим собой — бессмысленно.
- **Без GitHub Projects v2.** На этом масштабе это лишний оверхед: кастомные поля, которые никто не фильтрует, и доска, которую никто не открывает. Хватает списка Issues или `todo`.
- **Без knowledge-модуля**, пока не накопилось 5+ внешних источников (интервью, ресёрч).
- **Без выделения bounded context в отдельные крейты**, пока они спокойно живут модулями внутри `domain`.

Когда появится третий человек, вторая крупная часть продукта или первый enterprise-клиент — тогда смотри полный `framework.md`. Не раньше.

---

Фреймворк — это леса, не истина. Он окупается, если есть дисциплина поддерживать vault и уважение к накопленному контексту. Без этого — просто ещё одна папка устаревших доков. Сомнения и контрпримеры — норма: фреймворк адаптируется, а не копируется.
